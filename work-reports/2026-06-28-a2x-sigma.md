### Tokenizer (`tokenizer.rs`)
- `lex(input: &str) -> Result<Vec<Token>, LexError>` ‚Äî converts Œ£‚àû text to tokens
- Handles: boundary markers (‚ü¶‚üß‚ü¨‚ü≠), field separator (‚à∑), angle-bracket labels (‚ü®text‚ü©), multi-char protocol ID (Œ£‚àû), all 4 operator types, alphanumeric labels, colons
- Whitespace skipped between tokens

### Parser (`parser.rs`)
- `parse(tokens: &[Token]) -> Result<Vec<SigmaPacket>, ParseError>` ‚Äî tokens ‚Üí packets
- `parse_one_packet` ‚Äî parses a single ‚ü¶Œ£‚àû‚üß‚ü¨...‚ü≠ instruction
- Handles I/C/P/D fields with operator sequences and labels

### SigmaPacket (`packet.rs`)
- Field types: `IntentField`, `ContextField`, `PlanField`, `DataField`
- Context field supports both operators and label references
- Data field supports both operators and raw payload bytes

### SigmaProgram (`program.rs`)
- Instruction stream with label table and sub-programs
- Methods: `push`, `resolve_label`, `register_label`, `compose`, `compute_id`
- `compute_id` uses Blake3 to hash instructions + labels + sub-program names
- `ProgramMetadata` and `ProgramRef` types

### Display
- `Display` impl on `SigmaPacket` ‚Äî serializes back to canonical Œ£‚àû text form
- Proper separator handling (no trailing ` ‚à∑ `)

### Convenience API
- `parse_program(input: &str)` ‚Äî combined lex+parse+program
- `serialize_packet(packet: &SigmaPacket)` ‚Äî packet ‚Üí text

## Bugs Fixed During Implementation

1. **Tokenizer colon bug**: `:` was included in alphanumeric label character set, causing `I:` to tokenize as single label instead of two tokens
2. **Display trailing separator**: ` ‚à∑ ` was written after every field including the last
3. **compute_id**: Labels weren't actually hashed (empty loop)
4. **compose test**: Wrong label index expectations
5. **Clippy warnings**: unused vars, dead code, unnecessary mut
