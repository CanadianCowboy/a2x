## What Is A2X?

A2X (Agent-to-Anything) is a **three-layer programming language stack** designed
for AI agents:

| Layer | Name | Role |
|:-----:|------|------|
| **Œ£‚àû** | Sigma Infinity | Symbolic ISA ‚Äî Unicode-dense instruction packets |
| **Œ©** | Omega | Compiled latent representation ‚Äî pure tensors |
| **CCS** | CryoCore Cognitive Substrate | Runtime VM ‚Äî WorldGraph, StateField, MemoryTrace |

A2X is NOT a protocol (like MCP/A2A), NOT a framework (like LangChain), NOT a
binary format (like WASM). It's a **language** ‚Äî that AI agents write, compile,
and execute programs in.

## Key Types

### a2x-core
- `AgentId` ‚Äî string-based agent identifier
- `AgentType` ‚Äî enum: Orchestrator, Cli, Llm, Ccs, Omega, Entity
- `Capability` ‚Äî enum: Execute, FileSystem, Network, Shell, Probe, Custom(String)
- `Opcode` ‚Äî enum: Nop, Bind, Differentiate, Ground, Evolve, Reflect, Plan, Actuate, Jump, Branch, Call, Return, Fork, Merge, Halt, Custom(String)
- `ConceptVector` ‚Äî dense f32 embedding vector
- `ProgramId` ‚Äî Blake3 content-addressed program identifier
- `StateSnapshot` ‚Äî agent state for probing
- `Packet` ‚Äî unified message type: Raw(Vec<u8>) | Sigma(SigmaPacket) | Omega(OmegaPacket)

### a2x-sigma
- `SigmaPacket` ‚Äî single instruction: intent + context + plan + data fields
- `SigmaProgram` ‚Äî sequence of SigmaPackets with labels and sub-programs
- `IntentOp` ‚Äî enum: Lightning(‚ö°), Star(‚ú¶), Synthesis(‚ú£), Cancel(‚úï), Split(‚©´), Contradiction(‚üÅ), Delay(‚ßñ), Parallel(‚•Å), Merge(‚´¥)
- `PlanOp` ‚Äî enum: Sequential(‚•Ç), Branch(‚§ê), Descend(‚§à), Ascend(‚§â), Swarm(‚•Å), Merge(‚§ë)
- `DataOp` ‚Äî enum: Empty(‚åµ), GraphDelta(‚åÆ), Scalar(‚å≥), Vector(‚å±), Payload(‚å¥)

### a2x-ccs
- `CcsVm` ‚Äî the execution VM: WorldGraph + StateField + MemoryTrace + IP
- `VmStatus` ‚Äî enum: Running, Halted, Yield, Suspended, Fault(VmError)
- `WorldGraph` ‚Äî petgraph-based concept graph with label index
- `StateField` ‚Äî flat f32 array with named regions (belief, attention, temporal)
- `MemoryTrace` ‚Äî execution log of MemoryEntry records

## Conventions

1. **Plan first, code second** ‚Äî read the relevant sub-plan in `plans/` before writing code
2. **Sub-plan references** ‚Äî every file references its plan: `// See plans/03-ccs-vm.md ¬ß5`
3. **One concept per file** ‚Äî module `mod.rs` re-exports only
4. **Feature gates** ‚Äî optional deps behind named features; `serde` gate on serializable types
5. **No unwrap in library code** ‚Äî use `Result<T, E>` or `Option<T>` with proper error handling
6. **Tests for all changes** ‚Äî infrastructure gets unit tests; cognitive code gets property tests
7. **Work reports required** ‚Äî before committing, create `work-reports/YYYY-MM-DD-description.md`

## Common Tasks

### Adding a new Œ£‚àû operator
1. Add the operator symbol to `a2x-sigma/src/intent.rs` (or plan.rs / data.rs)
2. Add the Opcode mapping to `a2x-ccs/src/vm.rs::map_to_opcode()`
3. Add the operator implementation in `a2x-ccs/src/operators/`
4. Wire it into `CcsVm::step()` dispatch match
5. Add tests

### Adding a new agent type
1. Add variant to `AgentType` in `a2x-core/src/agent_id.rs`
2. Create `a2x-agents/src/new_agent.rs` implementing `Agent` trait
3. Register in `Gateway::register_builtin_agents()` and `a2x-cli/src/main.rs`
4. Add tests

### Adding a new dependency
1. Add to `[workspace.dependencies]` in root `Cargo.toml`
2. Add to the specific crate's `Cargo.toml` with `{ workspace = true }`
3. Never add to `a2x-core` (zero-dependency crate)
